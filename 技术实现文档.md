# 儿童学习陪伴小程序 · 技术实现文档

> 版本：V1.0  
> 日期：2026-05-21  
> 对应需求：`需求文档.md` V5.0  
> MVP 范围：1 套完整卡池 + 3 套预览主题

---

## 一、技术目标

第一版目标是先验证“规划作业 → 专注完成 → 充能 → 开包 → 收藏”的核心闭环。

技术实现优先保证：

- 每日任务和开包状态可靠，不重复发包。
- 抽卡结果由服务端生成，前端只负责展示。
- 收藏、合成、保底规则可追溯、可测试。
- 页面结构简单，便于后续扩展盲盒、家长端、更多主题。

---

## 二、技术选型

| 模块 | 选型 | 说明 |
|---|---|---|
| 小程序框架 | 微信小程序原生 | 降低依赖，方便使用云开发能力 |

## 四、页面实现

### 4.1 主页 `pages/home`

职责：

- 拉取今日任务、充能状态、当前主题、今日是否已开包。
- 展示充能袋、任务列表、添加任务入口、开包入口、收藏本入口、设置入口。
- 完成全部任务后引导进入开包页。

关键状态：

```js
{
  date: '2026-05-21',
  tasks: [],
  chargePercent: 0,
  allCompleted: false,
  packGenerated: false,
  packOpened: false,
  activeSeries: {}
}
```

### 4.2 添加任务页 `pages/task-edit`

职责：

- 创建或编辑当天任务。
- 校验任务名 `1~20` 字，预估时间 `5~90` 分钟，每天最多 8 项。
- 当日已开包后，不允许修改今日任务。

### 4.3 专注页 `pages/focus`

职责：

- 展示任务名、倒计时、长按完成按钮。
- 倒计时归零只播放轻提示，不自动完成。
- 长按完成后调用 `completeTask` 云函数。

注意：

- 计时器只作为前端体验，不作为完成判定依据。
- 小程序关闭后重新进入，任务状态以服务端为准。

### 4.4 开包页 `pages/pack-open`

职责：

- 进入页面时调用 `generateDailyPack` 或读取未展示的今日包。
- 按阶段展示开包动画。
- 用户翻完全部卡后调用 `openPack` 标记已展示，并跳转收藏本。

阶段：

1. 袋子颤动。
2. 光芒爆开，展示卡数。
3. 卡片轮廓与稀有度边框。
4. 逐张翻面。

MVP 可先实现 2-3 个关键阶段，保留组件接口用于后续增强。

### 4.5 收藏本 `pages/album`

职责：

- 展示当前已开放主题。
- 展示 50 个固定卡槽。
- 已获得卡显示缩略图，未获得卡显示灰色占位。
- 拥有数量 `>= 4` 的卡显示合成入口。

### 4.6 卡片详情 `pages/card-detail`

职责：

- 展示高清卡图、卡名、编号、稀有度、首次获得日期。
- SSR/SR 使用前端动效叠加稀有度效果。

### 4.7 主题选择 `pages/theme-select`

职责：

- 首次进入时选择当前主题。
- MVP 中只有 1 套可完整选择，其余 3 套可展示为预览。
- 后续版本开放 15 天冷却切换。

### 4.8 设置页 `pages/settings`

职责：

- 语音开关、音量。
- 抽卡概率公示。
- 新卡保底规则说明。
- 关于与合规说明。

## 六、云函数设计

### 6.1 `login`

输入：无。

输出：

```js
{
  openid,
  user
}
```

职责：

- 获取用户 `openid`。
- 若用户不存在则创建默认用户。

### 6.2 `getDailyState`

输入：

```js
{
  date
}
```

输出：

```js
{
  user,
  dailyTask,
  activeSeries,
  todayPack,
  collectionSummary
}
```

职责：

- 返回主页需要的全部状态。
- 如果当天任务记录不存在，返回空任务结构。

### 6.3 `saveTasks`

输入：

```js
{
  date,
  tasks: [{ task_id, name, estimated_minutes }]
}
```

规则：

- 任务数量 `1~8`。
- 任务名 `1~20` 字。
- 预估时间 `5~90` 分钟。
- 当天已生成基础包后不允许修改任务。

输出：

```js
{
  dailyTask
}
```

### 6.4 `completeTask`

输入：

```js
{
  date,
  task_id
}
```

职责：

- 将任务标记为完成。
- 重新计算 `completed_minutes` 和 `charge_percent`。
- 如果全部完成，更新连续完成天数，但不直接发包。

输出：

```js
{
  dailyTask,
  shouldGeneratePack
}
```

### 6.5 `generateDailyPack`

输入：

```js
{
  date
}
```

职责：

- 校验今日任务已全部完成。
- 校验今日未生成过基础包。
- 根据当前主题、概率、新卡保底生成卡包。
- 写入 `pack_records`。
- 同步更新 `collections`。
- 设置 `daily_tasks.pack_generated = true`。

输出：

```js
{
  pack
}
```

幂等要求：

- 如果今日基础包已生成但未展示，直接返回已有包。
- 不允许重复创建每日基础包。

### 6.6 `openPack`

输入：

```js
{
  pack_id
}
```

职责：

- 标记 `pack_records.opened = true`。
- 如果是每日基础包，同步设置 `daily_tasks.pack_opened = true`。

### 6.7 `synthesizePack`

输入：

```js
{
  series_id,
  source_card_id
}
```

规则：

- 用户拥有该卡数量必须 `>= 4`。
- 消耗 3 张，保留至少 1 张。
- 生成同主题合成包，包内 1~3 张。
- 合成包 SSR 概率 5%。

输出：

```js
{
  pack
}
```

### 6.8 `switchTheme`

MVP 可只做预留接口。

输入：

```js
{
  target_series_id
}
```

规则：

- 首次选择允许。
- 后续切换需要距离 `last_switch_date >= 15` 天。
- MVP 中 preview 主题不可切换为 active。

### 6.9 `updateSettings`

输入：

```js
{
  voice_enabled,
  voice_type,
  volume
}
```

职责：

- 更新用户设置。
- 可选写入 `settings_logs`。

## 七、抽卡算法

### 7.1 每日基础包卡数

| 卡数 | 概率 |
|---|---|
| 1 | 25% |
| 2 | 25% |
| 3 | 25% |
| 4 | 18% |
| 5 | 7% |

说明：原需求中 `2-3 张` 合计 50%，技术实现中拆成 2 张 25%、3 张 25%，便于实现和统计。

### 7.2 基础稀有度概率

| 稀有度 | 概率 |
|---|---|
| SSR | 3% |
| SR | 12% |
| R | 30% |
| N | 55% |

### 7.3 每日新卡保底

规则：

- 当前主题未集齐时，每日基础包至少包含 1 张未拥有卡。
- 如果普通概率结果没有新卡，则替换包内最后 1 张为未拥有卡。
- 替换时优先遵守当前缺失卡的稀有度分布，但可在后期保护阶段放宽。

### 7.4 后期保护

当当前主题收集进度 `>= 45/50`：

- 每 2 次每日基础包至少命中 1 张未拥有卡。
- 如果缺失卡中包含 SSR，允许保底直接命中缺失 SSR。

需要在 `users` 或独立统计中记录当前主题的后期保护计数。MVP 可通过查询最近基础包记录判断。

### 7.5 合成包算法

合成包规则：

- 卡数随机 1~3 张。
- SSR 概率提升到 5%。
- 不保证必出新卡。
- 不影响每日基础包上限。

### 7.6 推荐伪代码

```js
function generatePack({ openid, seriesId, source }) {
  const cardCount = source === 'daily'
    ? rollDailyCardCount()
    : rollSynthCardCount()

  const owned = getOwnedCardIds(openid, seriesId)
  const missing = getMissingCards(seriesId, owned)

  let cards = []
  for (let i = 0; i < cardCount; i++) {
    const rarity = rollRarity(source)
    cards.push(rollCardByRarity(seriesId, rarity))
  }

  if (source === 'daily' && missing.length > 0 && !containsNew(cards, owned)) {
    const guaranteed = pickGuaranteedMissingCard(seriesId, missing, openid)
    cards[cards.length - 1] = guaranteed
  }

  return markIsNew(cards, owned)
}
```

## 八、每日状态机

```text
EMPTY
  ↓ saveTasks
PLANNED
  ↓ completeTask 部分完成
IN_PROGRESS
  ↓ completeTask 全部完成
COMPLETED
  ↓ generateDailyPack
PACK_GENERATED
  ↓ openPack
PACK_OPENED
```

状态说明：

| 状态 | 含义 |
|---|---|
| EMPTY | 当天没有任务 |
| PLANNED | 已添加任务，尚未完成 |
| IN_PROGRESS | 至少完成 1 项，但未全部完成 |
| COMPLETED | 全部任务完成，尚未生成卡包 |
| PACK_GENERATED | 卡包已生成，可能还没展示完 |
| PACK_OPENED | 今日闭环完成 |

关键约束：

- `PACK_GENERATED` 后不可修改今日任务。
- `PACK_OPENED` 后不可再次生成每日基础包。
- 开包结果必须先入库，再播放动画。

## 九、权限与安全

- 所有写操作必须在云函数中校验 `openid`。
- 前端不得直接写 `collections`、`pack_records`、`daily_tasks.pack_generated`。
- 卡池配置由管理端或初始化脚本写入，普通用户不可修改。
- 抽卡概率和保底逻辑只在服务端执行。
- 云函数需对任务数量、时间、文本长度做二次校验。
- 现实奖励相关能力在 MVP 暂缓，避免未完成家长授权链路。

## 十二、测试清单

### 12.1 任务测试

- 添加 0 个任务时保存失败。
- 添加 9 个任务时保存失败。
- 任务时间小于 5 或大于 90 时保存失败。
- 已开包后修改任务失败。
- 完成部分任务后充能比例正确。

### 12.2 开包测试

- 未完成全部任务时不能生成每日包。
- 完成全部任务后可生成每日包。
- 同一天重复调用 `generateDailyPack` 返回同一个包。
- 开包中断后重新进入展示同一个包。
- 未集齐主题时每日包至少包含 1 张新卡。

### 12.3 收藏测试

- 新卡写入 `collections`。
- 重复卡 `count + 1`。
- 已获得卡显示卡面，未获得卡显示灰影。
- 收藏进度按唯一卡数量计算。

### 12.4 合成测试

- `count = 3` 不显示合成。
- `count = 4` 显示合成。
- 合成后原卡 `count - 3`。
- 合成包写入 `pack_records`，且 `source = synth`。

### 12.5 设置测试

- 关闭语音后不播放语音。
- 音量修改后下次播放生效。
- 设置页展示概率和保底说明。
